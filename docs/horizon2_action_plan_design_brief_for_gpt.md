# H=2 Action EV 方案展示详设 Brief

## 背景

当前 `position_strategy_efficiency_rows(..., horizon=2)` 已经在数学上计算两步 lookahead：

- 顶层 action `a1`；
- 每个 outcome 后递归选择后续最优 action `a2(state_after_outcome)`；
- 返回的是 `max_a Σ p(outcome | a) * V1(next_state)` 的精确期望。

UI 目前的问题不是 EV 没算两步，而是结果表达像“只推荐第一步”，并且“代表搭配”容易误导：

- H=2 推荐卡只显示一个 action，看不出第二步策略；
- 随机位置 action 是 1-6 固定位置的概率混合，不存在唯一典型搭配；
- 随机 action 当前如果给一个单独代表新盘/代表搭配，会让用户误以为随机结果确定落在某个位置；
- 用户希望看到“方案”，即第一步怎么做、命中不同结果后第二步怎么做、最后搭配如何满足 4+2 / 2+2+2 硬约束。

## 不变约束

1. 不改变默认策略。
2. 不引入快速预览。
3. 不引入近似推荐。
4. 不引入 Monte Carlo。
5. 不展示 partial 推荐。
6. 不默认启用 state-DP。
7. 不默认启用 ProcessPool 并行。
8. H=2 仍为完整概率分布精确计算。
9. 当前 EV 排序口径暂不改，只改“方案解释/审计输出”。

## 需要 GPT 详设的问题

请设计一个 H=2 Action EV 的“条件方案展示”数据结构和 UI 呈现方式，使用户能审计：

1. 顶层推荐 action 为什么是这个；
2. 如果顶层 action 是固定位置，代表分支下第二步做什么；
3. 如果顶层 action 是随机位置，1-6 每个位置命中后的第二步策略分别是什么；
4. 每个分支最终如何搭配其余位置来满足套装硬约束；
5. 哪些输出是“精确期望”，哪些只是“代表分支/审计样例”。

## 建议语义

### 1. action 类型语义

- 固定位置：
  - 是基础 action 行；
  - 有明确目标位置；
  - 可以展示代表 outcome、代表后续 action、代表最终搭配。

- 随机位置：
  - 是六个固定位置 action 的概率混合；
  - 不应展示唯一代表搭配；
  - 应展示“条件方案树”：
    - 命中 1号位 -> 后续最优 action；
    - 命中 2号位 -> 后续最优 action；
    - ...
    - 命中 6号位 -> 后续最优 action。

- 固定主属性 / 固定副属性：
  - 不应直接与随机基准混在一起解释；
  - 应分别和上一级 action 比较：
    - 锁主属性 vs 同位置固定位置；
    - 锁副属性 vs 同位置锁主属性。

### 2. H=2 “方案”不是单一路径

H=2 的方案应该表达成 policy，而不是单条 path：

```text
推荐第一步：随机 折枝剑歌

条件第二步：
- 若第 1 步命中 1号位：下一步固定 6号位 折枝剑歌
- 若第 1 步命中 2号位：下一步固定 5号位 折枝剑歌
- ...
```

固定位置 action 可以额外给一个代表路径：

```text
推荐第一步：固定 6号位 折枝剑歌
代表命中：6号位折枝剑歌
第二步代表后续：固定 5号位 折枝剑歌
代表最终搭配：云岿如我4 + 折枝剑歌2
```

但必须标注：

```text
代表路径只用于审计；真实 H=2 EV 已对所有 outcome 加权。
```

## 建议输出字段

在 Action EV row 上新增审计字段，不替换现有 EV 数值字段：

```python
{
    "方案类型": "单步" | "代表路径" | "条件策略",
    "第一步 action": "...",
    "第二步策略摘要": "...",
    "条件分支": [
        {
            "条件": "第1步命中 1号位",
            "条件概率": 0.1666667,
            "代表新盘": "...",
            "第二步 action": "...",
            "第二步原因": "...",
            "代表最终搭配": "...",
            "套装约束": "满足..."
        }
    ],
    "代表路径说明": "仅代表高价值/高概率分支；EV 已完整枚举",
}
```

对随机位置：

- `方案类型 = 条件策略`
- `代表最终搭配` 不应作为唯一表格展示；
- UI 可展示条件分支表。

对固定位置：

- `方案类型 = 代表路径`
- 可以继续展示“推荐调律后代表搭配”表；
- 但标题应写“代表分支搭配”，而不是“预期唯一搭配”。

## UI 建议

结果页建议拆成 3 个子区：

1. 推荐卡：
   - 推荐第一步 action；
   - horizon；
   - 计算口径；
   - 方案类型；
   - 主要 EV 指标。

2. H=2 方案：
   - 固定 action：展示代表路径 + 代表最终搭配；
   - 随机 action：展示 1-6 条件分支表，不展示唯一搭配。

3. Action EV 明细：
   - 展示所有 action 行；
   - 固定位置行作为基础审计；
   - 随机行标为“混合结果”。

## 算法实现建议

不要重复计算完整 EV。复用已有函数：

- `_aggregated_action_outcomes_for_spec`
- `_best_followup_spec`
- `_best_combo_rows`
- `_new_rows_between`
- `_rows_present_in_combo`
- `_combo_without_rows`

为每个顶层 action 构造 explain-only 的 policy summary：

1. 对顶层 action 的 outcome 分布聚合；
2. 对每个 outcome state 调用 `_best_followup_spec(..., horizon=1)`；
3. 按分支类型汇总：
   - 固定位置：取高价值/高概率代表分支；
   - 随机位置：按命中 position 分组，各组保留代表后续 action；
4. 输出解释字段，不影响 `_sort_vector` 和 EV 排序。
